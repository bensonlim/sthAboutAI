## 5. 代码开发规范

### 5.1 DTO设计规范

1. **查询入参规范**：所有查询接口的入参必须生成新的DTO类，禁止直接使用实体类或其他业务DTO作为查询入参。

2. **查询出参规范**：所有查询接口的出参必须生成新的DTO类，禁止直接返回实体类。出参DTO应仅包含前端所需字段，避免返回冗余数据。

3. **DTO命名规范**：
    - 查询入参DTO命名格式：`{业务模块名称}QueryDTO`，如：`MtMakeOrderQueryDTO`
    - 查询出参DTO命名格式：`{业务模块名称}ResponseDTO`，如：`MtMakeOrderResponseDTO`
    - 复杂查询场景可在命名后添加序号，如：`MtMakeOrderQueryDTO01`

### 5.2 Mapper开发规范

1. **参数注解规范**：当Mapper接口方法的查询对象是某个DTO时，必须使用`@Param("dto")`注解明确指定参数名称，便于在XML映射文件中引用。

2. **XML映射规范**：在XML映射文件中引用DTO参数时，必须使用`#{dto.fieldName}`格式，如：`#{dto.makeOrderNum}`。

3. **SQL编写规范**：
    - 所有查询必须使用参数绑定，禁止使用字符串拼接
    - 复杂查询必须使用动态SQL标签（如`<if>`, `<foreach>`）
    - 字段名必须使用别名，避免表连接时字段冲突

### 5.3 Service开发规范

1. **分页逻辑迁移规范**：所有使用`PageHelper.doPageAndSort`的分页逻辑必须从Service层迁移到Repository层，确保分页处理与数据访问逻辑紧密结合。

2. **方法命名规范**：Service层方法名应明确反映业务功能，使用动词+名词格式，如：`getMakeOrderList`、`queryMakeOrderByNum`。

3. **参数传递规范**：Service层方法必须将`tenantId`作为第一个参数接收，并向下传递给Repository层。

4. **实体到DTO转换规范**：当Repository查询结果为实体对象，而接口返回需要DTO对象时，必须在Service层使用MapStruct进行转换，确保数据转换逻辑集中管理和高效执行。

    - 转换逻辑必须使用MapStruct框架实现，禁止手动编写大量setter/getter代码
    - 转换接口应命名为`{EntityName}Mapper`，如：`MtMakeOrderMapper`
    - 转换方法应明确反映转换方向，如：`toDTO`、`toDTOList`

### 5.4 Controller开发规范

1. **路径参数规范**：查询接口中使用`@PathVariable`注解的路径参数，若参数名为`organizationId`，必须修改为`tenantId`。

2. **参数传递规范**：`tenantId`必须作为第一个参数向下传递，贯穿整个调用链（Controller → Service → Repository → Mapper）。

3. **方法命名规范**：Controller层方法名应明确反映业务功能，使用动词+名词格式，如：`getMakeOrderList`、`queryMakeOrderByNum`。

4. **注解使用规范**：
    - 查询接口可以使用`@GetMapping`或`@PostMapping`注解
    - 新增/修改/删除接口必须使用`@PostMapping`注解
    - 禁止使用`@RequestMapping`的默认请求方式，必须明确指定请求方法

### 5.5 实体类使用规范

1. **禁止新增字段规范**：在新增接口时，严禁往实体类（Entity）中增加字段。实体类应仅包含数据库表对应的字段，保持与数据库表结构的一致性。

2. **字段扩展原则**：如果接口需要返回额外的字段或接收额外的参数，必须通过以下方式实现：
    - 查询接口：创建新的查询入参DTO和查询出参DTO
    - 新增/修改接口：创建新的请求DTO和响应DTO
    - DTO中可以包含业务所需的额外字段，但不得修改实体类

3. **实体类职责**：实体类仅用于与数据库表的映射，不应承担业务逻辑或接口参数传递的职责。

4. **代码示例**：
    - 错误做法：在`MtMakeOrder`实体类中添加`customerName`字段用于查询返回
    - 正确做法：创建`MtMakeOrderResponseDTO`，在其中添加`customerName`字段，通过关联查询获取数据后填充到DTO中

5. **违规后果**：违反此规范将导致：
    - 实体类与数据库表结构不一致，可能引发数据持久化错误
    - 影响其他使用该实体类的功能，造成不可预期的副作用
    - 降低代码可维护性，增加系统复杂度
